# AI开发平台

## 项目概述

AI开发平台是一个智能的软件开发辅助工具，帮助产品经理跟架构师进行需求分析、系统设计、文档生成和开发指导。

### 上下文工程
上下文工程（Context Engineering）是一门专注于大语言模型信息载荷的系统性设计、组织和优化的新兴学科。简单来说，它就是教会AI如何更好地"理解背景"、"记住重要信息"、"智能组织知识"的技术。

上下文工程具有以下五个核心特征：

1. 系统性方法：不只关注单次对话，而是构建完整的信息生态系统
2. 动态优化：能够根据任务需求实时调整信息内容和结构
3. 多模态融合：支持文本、图像、音频等多种信息类型的统一处理
4. 状态管理：具备长期记忆能力，能够维护会话状态
5. 智能组装：通过算法自动选择和组合最相关的信息

## 功能特性

### 核心功能
- 💬 **交互式对话引导**: 通过连续的、有引导性的问题，来澄清和挖掘需求的细节。
- 🧠 **智能需求分析**：AI自动分析用户需求，提取核心功能和业务流程,采用了一种被称为EARS（Easy Approach to Requirements Syntax，简易需求语法）的方法论，来撰写需求。
- 📊 **可视化与代码双向同步**：自动生成PUML架构图,用户可以直接在生成的图表上进行拖拽、修改，系统会自动更新背后的PUML代码；反之，修改PUML代码，图表也会实时渲染。
- 📝 **知识库驱动（RAG）**：内置一个可扩展的知识库，包含常见的架构模式、设计原则、组件库等，AI的生成将优先从知识库中检索信息，减少“幻觉”，提高产出质量。
- ⚙️ **用户自定义AI配置**：支持OpenAI、Claude、Gemini等多种AI提供商

## 项目解决问题
当遇到大项目,AI会面临以下几个问题：
1. 注意力稀释： 在极长的上下文中，AI很难判断哪些是当前任务最核心的信息，早期的、可能已经过时的指令会干扰它对当前需求的理解。
2. “灾难性遗忘”： 为了处理新的指令，模型可能会“忘记”或忽略掉一些早期的关键细节或约束。
3. 错误累积： 如果AI在早期生成了一个小错误，而您没有发现，在后续的开发中，AI会基于这个错误的认知继续构建，导致问题像滚雪球一样越滚越大。
4. token限制： 即使是拥有超长上下文窗口的模型，也存在物理上限。超出限制后，最早的信息会被截断，导致信息丢失。

要解决这个问题，核心思想是**“化整为零，分而治之” (Divide and Conquer)**。您需要从一个“项目经理”的角色，转变为一个“技术总监”或“架构师”的角色，将与AI的协作模式进行优化。不要把AI看作一个无所不能、一次性就能完成任务的黑箱，而是把它看作一个能力极强但需要明确指导的初级程序员。


## 项目功能交互

### 第一阶段：产品设计与架构阶段 (What & How)

这个阶段的核心目标是：将用户的初步想法，转化为清晰、可视化的产品设计蓝图。

#### 步骤 1：交互式需求探索 (Interactive Requirement Discovery)

用户操作： 输入一段关于项目的初步、模糊的想法。

平台AI动作：

- 启动引导式对话： AI作为虚拟产品专家，不会立即输出长篇大论，而是通过一系列关键问题来澄清、挖掘和补全需求细节（例如：“您的用户群体是谁？”“需要哪些登录方式？”“核心的盈利模式是什么？”）。

- 实时总结： 在对话过程中，AI会动态地将澄清后的需求点，结构化地整理成一份详细的需求描述。

- 核心产出： 一份详尽、结构化的 《原始需求纪要 (PRD V0.1)》。

- 用户体验： 整个过程就像与一位资深的产品经理进行头脑风暴。

#### 步骤 2：功能列表生成与确认 (Feature List Generation & Confirmation)

用户操作： 查看AI根据《原始需求纪要》生成的功能列表。

平台AI动作：

- 生成功能清单： AI将需求纪要自动转换为一个分模块的功能列表（Feature List），并对每个功能进行简要描述。

- 识别核心功能： AI会根据行业经验和需求描述，自动推荐并高亮出“核心功能”。

核心产出：

- 一份可在线编辑、可调整优先级的 《功能列表》。

- 一份突出显示的 《核心功能模块描述》。

用户体验： 用户可以像使用在线文档一样，自由增、删、改功能列表，并最终确认范围。所有修改都将保存在数据库中。

#### 步骤 3：核心功能逻辑可视化 (Core Feature Logic Visualization)

用户操作： 在功能列表中，选择一个或多个“核心功能”，点击“生成逻辑图”。

平台AI动作：

- 选择图表类型： AI会建议最适合该功能的图表类型（例如，用户注册流程适合用“时序图”，订单状态流转适合用“活动图”）。

- 生成PUML代码： AI根据功能描述，自动生成描述业务逻辑的PUML代码。

- 核心产出： 针对每个核心功能的可视化 PUML逻辑交互图（时序图、活动图等）。

- 用户体验： 用户可以在线编辑PUML代码，右侧实时预览图表变化，所见即所得。图表和代码保存在数据库中，与对应功能关联。

#### 步骤 4：系统架构设计 (System Architecture Design)

用户操作： 在确认了整体功能和核心逻辑后，点击“生成系统架构”。

平台AI动作：

- 模块识别： AI分析完整的功能列表，自动识别出系统需要哪些大的技术模块（例如：用户服务、订单服务、API网关、支付网关、数据库、缓存等）。

- 生成架构图： AI从内置的架构模式知识库中，选择最合适的模式（如微服务、单体架构），并生成描述模块间关系的 C4架构图 的PUML代码。

- 核心产出： 一份可视化的 PUML项目架构图 和 模块划分说明。

- 用户体验： 同样支持在线编辑和实时预览，帮助技术团队在开发前就对整体结构达成共识。所有内容保存在数据库中。

### 第二阶段：工程实现与质量保障阶段 (Implementation & QA)

这个阶段的核心目标是：将设计蓝图，转化为可执行、可测试的工程化文档。

#### 步骤 5：开发任务拆解 (Development Task Breakdown)

用户操作： 点击“生成开发计划”。

平台AI动作：

- 任务分解： AI扮演敏捷教练的角色，将功能列表中的每一个功能，自动分解为更小的开发任务（User Stories 或 Sub-tasks）。

- 估算与排序： AI可以根据任务的复杂度，初步估算工作量，并建议一个符合逻辑依赖关系的开发顺序。

- 核心产出： 一份结构化的 《开发步骤/任务清单》，可直接导入到Jira、Trello等项目管理工具。

#### 步骤 6：API接口文档生成 (API Specification Generation)

用户操作： 针对某个开发任务（特别是涉及后端服务的），点击“生成API文档”。

平台AI动作：

- 接口定义： AI结合架构图和逻辑交互图，自动为模块间的调用生成详细的API接口定义，包括请求路径、方法、参数、返回数据结构等。

- 格式化输出： AI可以将接口定义输出为标准格式，如 OpenAPI (Swagger) 3.0。

- 核心产出： 一份可在线编辑、可导出的 《API接口文档》，保存在数据库中。

#### 步骤 7：测试用例生成 (Test Case Generation)

用户操作： 针对任意功能、API接口或底层模块，点击“生成测试用例”。

平台AI动作：

- 多维度测试： AI可以根据用户选择的维度，生成不同类型的测试用例：

- 单元测试： 针对底层工具模块或某个函数的代码片段。

- API接口测试： 针对API文档，生成请求脚本和断言。

- 端到端(E2E)测试： 针对核心交互流程，生成BDD（行为驱动开发）风格的测试场景描述（Gherkin语言）。

- 核心产出： 一份覆盖全面的 《单元测试、接口测试、E2E测试用例集》，保存在数据库中。


### AI配置功能 ✨ **NEW**
系统已升级支持用户个性化AI配置：

## 技术栈

### 后端
- **语言**：Go 1.23
- **框架**：标准库 net/http
- **数据库**：MySQL 8.0
- **AI集成**：OpenAI API、Google Gemini API、Claude API

### 前端
- **框架**：React 18 + TypeScript
- **构建工具**：Vite
- **UI组件**：Ant Design
- **状态管理**：Zustand
- **样式**：Tailwind CSS

## 快速开始

### 环境要求
- Go 1.23+
- Node.js 18+
- MySQL 8.0+

### 启动步骤

1. **启动后端服务**
```bash
go run cmd/server/main.go
```

2. **启动前端服务**
```bash
cd web
npm install
npm run dev
```
